# Yomi-Tame Hiroba 自動ポイ活ツール

Android端末をPCからUSB制御し、[ヨミタメ広場](https://hiroba-sp.yomu-point.jp/magazineList/) の記事を自動で読み進めてdポイント（スタンプ）を全自動で獲得するスクリプトです。

## 注意事項
- 3日くらい連続で最大数まで動かしたら，その次の日から記事が読めなくなりました．ご注意ください．
- 2,3日何もしなかったら，もとに戻り，また記事を自動で読むことができるようになりました．

## 特徴
- **超軽量動作**: Playwright などの重いブラウザ自動化ライブラリを使用せず、Node.jsの標準機能とADBコマンドのみで完結させています．
- **効率的な数珠つなぎ処理**: 記事一覧に戻ることなく、「次の記事を読む」ボタンを経由して最大150記事まで一気に読み進めます。
- **スマートな終了判定**: 読むべき記事がなくなると自動で停止します。

## 動作確認済み環境
1. **PC側**
   - **OS**: Windows 10 Pro (10.0.19045)
   - **Node.js**: v24.11.1
   - **ADB**: Android Debug Bridge version 1.0.41 (Version 36.0.2-14143358)
2. **スマホ側**
   - **端末OS**: Android OS 13 
   - **[Braveブラウザ](https://play.google.com/store/apps/details?id=com.brave.browser)**: インストール済みが前提

## 前提条件
本ツールを使用するには、以下の環境が必要です。

1. **PC側**
   - [Node.js](https://nodejs.org/) がインストールされていること。
   - [Android SDK Platform-Tools](https://developer.android.com/studio/releases/platform-tools) (ADB) がインストールされ、システムの環境変数 (PATH) に設定されていること。

2. **Android端末側**
   - 開発者向けオプションから **USBデバッグ** が有効になっていること。
   - USBケーブル接続時に表示されるダイアログで **「ファイル転送（MTP）」を許可** していること。
   - Brave ブラウザ（または Chrome 等の標準ブラウザ）がインストールされており、ヨミタメ広場にログイン済みであること。
      - 広告を読み込まないことで，より安定した動作を実現させています．

## 使い方

1. Android端末をPCにUSBケーブルで接続します。
2. 端末側で対象のブラウザ（Brave等）を開き、**事前に「ヨミタメ広場」へログインした状態**にしておきます。
3. **端末の画面ロックを解除し、画面をオン（アクティブ状態）にしておきます。**
   （※画面が暗くなっている状態やロック画面では、タップ等の自動操作が反映されません）
4. コマンドプロンプトやPowerShellを開き、接続をテストします。
   ```powershell
   adb devices
   ```
   **実行結果のサンプル：**
   ```text
   List of devices attached
   1234567890abcdef    device
   ```
   ※「device」と表示されればOKです。「unauthorized」や「offline」の場合はスマホ画面の許可ダイアログをOKするか、ケーブルを挿し直してください。
5. このフォルダに移動し、オブジェクト指向版スクリプトを実行します。
   ```powershell
   node launch_oop.js
   ```
6. あとはスマホの画面が自動で動き、スタンプを獲得し続けるのを眺めるか、放置してください。

### 【参考】Playwright版 (`launch.js`) を使用する場合
純粋ADB版ではなく、Playwright（ブラウザ自動化ライブラリ）を経由して実行する元のスクリプトも同梱されています。こちらを使用する場合は、初めに依存ライブラリのインストールが必要です。

1. プロジェクトフォルダで初期化とインストールのコマンドを実行します。
   ```powershell
   npm init -y
   npm install playwright
   ```
2. スクリプトを実行します。
   ```powershell
   node launch.js
   ```

### 【参考】テスト用スクリプト (`launch_test.js`) 
Playwrightが対象のAndroid端末で正常に動作するかどうか（ブラウザを直接制御できるか）を判定するためのスクリプトです。

1. ターミナルで実行します。
   ```powershell
   node launch_test.js
   ```
2. スクリプトが端末のChromeを強制終了し、新たにGoogleのトップページ（`https://www.google.com`）を開く挙動をします。
   - **※もしここでエラーが発生し、Googleのトップページ画面が開けない場合は、その端末ではPlaywrightによる高度な制御がブロックされています。その場合はその端末は基本ADBによる操作しかできないと考えてください．**

## 設定の調整
スクリプト冒頭の `CONFIG` オブジェクトを書き換えることで、動作を微調整できます。
ファイル名: `launch_oop.js`

```javascript
const CONFIG = {
    // ... 
    COORDS: {
        SECOND_ARTICLE: { x: 540, y: 1800 }, // ※最初の記事一覧画面で記事をタップする座標
        CONTINE_READING: { x: 540, y: 2000 }, // 「続きを読む」もしくは「スタンプGET」ボタンの位置（座標）
        NEXT_ARTICLE: { x: 792, y: 1232 } // 「次の記事を読む」ボタンの位置（座標）
    },
    // ...
};
```
### 注意: タップ座標の調整について
本ツール（オブジェクト指向版）は、Playwrightのようにブラウザ内部の要素（HTML）を直接解析してクリックする「高度なプログラム制御」を使っていません。Playwrightでブラウザを制御できなかったためです．そのため、**特定の操作（リストの一番最初の記事を選択する際や、スクロールの開始・終了位置など）を画面の「絶対座標（X, Y）」に依存して行っています。**

お使いのスマホの画面解像度やアスペクト比によっては、スクリプトが想定している場所をタップできず、最初の記事が開けないなどの期待通りの動作をしない可能性があります。うまく動作しない場合は、`COORDS` の各座標をご自身の端末に合わせて書き換えていただく必要があります。
